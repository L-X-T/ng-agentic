Refactoring a Dusty Angular Component

The payoff lab. You'll take a real, overgrown Angular component – the kind with any everywhere, lifecycle hooks, *ngIf, and an 800-line file – and rebuild it into something small, modern, and signal-based. You won't freestyle it: you'll follow a repeatable 7-step blueprint, let the AI do the heavy lifting, and review every change like it came from a junior on your team.

Angular v22 Package manager pnpm 10 Skill ng-refactor Est. time 90–150 min Level advanced

About this lab

This is Lab 04 – where the agentic setup from Lab 01 and the skill practice from Lab 02 finally earn their keep. Over years, components collect features, special cases, and workarounds until responsibilities blur between UI, state, and business logic. The result is a dusty component: high mental load, risky to change, slow to onboard to – and, crucially, harder for an AI agent to grasp than for a human. Good names, small functions, and precise types reduce both misunderstandings and hallucinations.

You'll fix that with the ng-refactor skill, which encodes a 7-step blueprint: Analyse → Declutter → Update → Modernize → Type → Refactor → Review. Each step is verified by the same gauntlet (build → lint → format → test) and paused at a clean checkpoint for you to review and commit before the next. The full written playbook lives in REFACTORING.md; this lab turns it into a hands-on run.

Why this matters

AI makes rewriting fast – which is exactly why it's dangerous without guardrails. A blueprint turns "rewrite this component" (a recipe for an unreviewable 800-line diff) into a sequence of small, verifiable moves. AI accelerates the rebuild; the architecture decisions stay human.

Learning objectives

  • Drive a legacy Angular component through a repeatable, verifiable refactoring blueprint.
  • Use the ng-refactor skill – and pair it with ng-migrate for the official schematics.
  • Replace any, lifecycle hooks, and old patterns with strict types, signals, inject(), control flow, and OnPush.
  • Apply AI guardrails: plan first, small reviewable steps, keep comments & domain logic, review before commit.
  • Prove each step with the gauntlet and let the linter act as your checklist.

Prerequisites

  • You've completed Lab 00 and Lab 01 – the workspace runs and the agentic tooling is in place (Prettier, ESLint, AGENTS.md, MCP servers, .aiignore).
  • The Angular MCP is connected so the agent works from current docs.
  • An AI coding agent that can read this repo and run commands (Claude Code, Junie, Codex, Cursor, …).
  • A dusty component to work on (see "Before you begin").
  • A clean Git working tree (git status) – the blueprint checkpoints after every step.
Before you begin – pick your target

Choose one overgrown component to carry through all seven steps. Start with legacy code in your own project from Lab 00. If you do not have a suitable legacy codebase available, use this workspace as the fallback:

  • Your own project (from Lab 00) – the primary path: pick its messiest component.
  • This repo – fallback only: src/app/components/table/ is a genuinely complex, multi-file feature.
  • The workshop example – another fallback: a deliberately dusty component at L-X-T/ng-days-refactoring-ai.

If you use a fallback target, expect the first lint or test run to expose existing debt. That is the starting point for the exercise: record the failures, add a behavior safety net, and only then start production refactoring.

How to work through this lab

Each step has a Goal, some background, a Your task box, a collapsible Solution with example prompts and commands, and a Checkpoint. This time the agent does most of the typing – your job is to steer and review. Start each step by invoking the skill: 

promptFollow the ng-refactor skill. We're on step N (<name>) for <path/to/component>.
Plan it, ask me anything unclear, then implement – small diffs only.

The blueprint – 7 steps

  1. Analyse – understand it, plan it, record the baseline
  2. Declutter (Entschlacken) – delete what's dead
  3. Update (NG Update) – modern toolbox
  4. Modernize (Modernisieren) – signals, control flow, OnPush
  5. Type (Typisierung) – kill any
  6. Refactor – shrink, split, structure
  7. Review – confirm the gains, then commit

1 Analyse

Goal: understand the component, write a plan, record the current baseline, and stand up a safety net.

Refactoring without a plan and without tests is gambling. First, read the component and its template, styles, and existing tests together. Name the single responsibility it should have; everything else is a candidate to extract or delete. Then make sure its behavior is pinned down by a test before you touch production code.

Your task

  1. Have the agent analyse the component and list its smells: LoC, lifecycle hooks, any/weak types, old patterns (*ngIf, @Input(), ngClass), template logic, duplication, unclear names.
  2. Ask it to propose an ordered plan following the blueprint – and to ask you clarifying questions before changing any code.
  3. Establish a safety net: confirm or add a behavior test, then run the gauntlet and record whether the baseline is green or already red.
Show solution – analysis prompt & baseline 
promptFollow the ng-refactor skill, step 1 (Analyse), for
src/app/components/table/table.component.ts (+ html/scss and any existing tests).

Its job is: render a paginated, sortable data table.
Analyse it, list the smells, and propose an ordered 7-step plan with the
expected win per step. Ask me clarifying questions first – DO NOT change code.

Pin behavior with an e2e test if coverage is thin (see the create-e2e-tests skill), then run the baseline. If it is red in a fallback target, write down the exact failures before changing production code:

shellgit status            # must be clean – the blueprint checkpoints per step
pnpm build && pnpm lint && pnpm format && pnpm test   # the gauntlet
Plan first, then implement

Resist the urge to let the agent start editing during analysis. A good prompt ends with "ask me clarifying questions first – don't change code yet." The quality of the spec up front decides the quality of everything after.

Checkpoint: you have a written plan, answered the agent's questions, a behavior safety net, and a green or explicitly documented red baseline. No production code changed.

2 Declutter (Entschlacken)

Goal: remove everything dead – pure subtraction, the lowest-risk change there is.

Before you modernize or restructure anything, take out the trash. Smaller surface area means every later step has less to reason about – for you and for the agent.

Your task

Have the agent delete dead code, unreachable branches, unused imports/inputs/outputs/fields/methods, and commented-out blocks – but keep meaningful comments and domain logic. Remove redundant SCSS. Then verify the gauntlet.

Show solution – declutter prompt 
promptng-refactor step 2 (Declutter) for the same component.
Remove dead code, unused members and imports, and commented-out blocks.
KEEP all meaningful comments and any domain logic. Drop redundant SCSS.
Behavior must not change. Show me the diff.
shellpnpm lint --fix && pnpm build && pnpm test
Keep the meaning

"Dead code" is not "code I don't understand." If a block looks pointless but encodes a domain rule, leave it and ask. The guardrail is explicit: never delete existing comments or domain logic.

Checkpoint: only live code remains, behavior is unchanged, and the gauntlet is green. The diff is small and obviously safe.

3 Update (NG Update)

Goal: make the modern toolbox available – a workspace-level step, run once, not per component.

You can't migrate to APIs your Angular version doesn't have. Bring Angular and its dependencies up to date so Step 4's schematics and modern APIs exist, and let the version's automatic migrations run.

Your task

  1. Update Angular and the CLI/ESLint packages; review the automatic migrations they apply.
  2. Bring other npm packages along – and handle security while versions move (CSP, Trusted Types, 3rd-party packages).
Show solution – update commands 
shellpnpm ng:update     # ng update @angular/cli @angular/core angular-eslint @angular-eslint/builder
# or, in an Nx workspace:
nx migrate latest && nx migrate --run-migrations

Review what the automatic migrations changed, then verify:

shellpnpm build && pnpm lint && pnpm test

For cadence and the automatic-migration list, see modern-angular.md.

Cadence

Keep Angular current: at least 2× a year for majors, ideally every couple of months for minors. Staying close to latest makes every future refactoring cheaper.

Checkpoint: Angular is current, automatic migrations are applied and reviewed, dependencies are updated, and the gauntlet is green.

4 Modernize (Modernisieren)

Goal: adopt modern Angular – official schematics first, then the changes they don't cover.

Prefer Angular's official migration schematics over hand-edits: they rewrite deterministically and update every reference (templates, tests, call sites). Delegate them to the ng-migrate skill, one migration per checkpoint. Then do the manual modernization the schematics can't.

Your task

  1. Run the schematics one at a time, verifying after each: standalone, control flow (@if/@for), inject(), signal input()/output()/queries, self-closing tags, ngClass/ngStyle → bindings, lazy routes, unused-import cleanup.
  2. Replace lifecycle hooks: move setup into the field initializer / constructor; takeUntilDestroyed() for manual subscriptions; effect() / afterNextRender() / afterRenderEffect() instead of ngAfterViewInit / ngOnChanges. Best of all: no hook.
  3. Switch to ChangeDetectionStrategy.OnPush; use computed() for derived state.
Show solution – migrate & modernize 
promptUse the ng-migrate skill to run the official schematics for this component,
one migration per checkpoint, gauntlet-green each time:
standalone → control-flow → inject → signal-input → signal-queries →
outputs → self-closing-tag → ngclass-to-class → ngstyle-to-style →
cleanup-unused-imports.
promptNow ng-refactor step 4: remove the remaining lifecycle hooks.
Move ngOnInit setup into field initializers; convert manual subscriptions to
takeUntilDestroyed(); replace ngOnChanges with computed() off signal inputs.
Set changeDetection to OnPush. Small diffs, show me each.

Mapping table and rule-by-rule notes: modern-angular.md.

One migration, one checkpoint

Don't let the agent chain ten migrations into one diff. Each schematic gets its own gauntlet and its own review – that's how you keep a green, bisectable history when something breaks.

Checkpoint: chosen migrations applied (each gauntlet-green and checkpointed), no replaceable lifecycle hooks remain, and the component is OnPush.

5 Type (Typisierung)

Goal: turn types into a safety net – the first real lever in legacy code.

Now that structure and APIs are modern, make the compiler your ally. Replacing fuzzy types with explicit ones surfaces hidden bugs and makes the structural cut in Step 6 safe.

Your task

Have the agent replace every any, add explicit return and parameter types, prefer unknown over any and type over interface, and extract small precisely-named types into their own files. Precisely type the inputs and outputs.

Show solution – typing prompt & check 
promptng-refactor step 5 (Type) for this component.
Remove every `any`; add explicit return and parameter types.
Prefer `unknown` over `any` and `type` over `interface`.
Extract non-trivial types into their own files (Rule of One).
Type the inputs/outputs precisely. Don't change behavior.

The linter is the checklist – these must be clean:

shellpnpm lint   # no @typescript-eslint/no-explicit-any, explicit-function-return-type
Before / after

items: any[] + sel: any + save(data: any) becomes an OrderItem type in its own file, with selectItem(item: OrderItem) and an early-return guard. Same behavior – but the compiler now guards every use, and the names carry meaning. See clean-code-and-structure.md.

Checkpoint: no any remains, no-explicit-any and explicit-function-return-type are clean, and the gauntlet is green.

6 Refactor

Goal: the real structural work – small functions, clear ownership, readable top-to-bottom.

With dead code gone, APIs modern, and types tight, the structural changes are now safe and obvious. Work in small, reviewable diffs – and make every change earn its keep.

Your task

  • Shrink functions. Split big event handlers (validation + mapping + state + UI side effects) into small, intent-revealing functions (buildRequest, validateInput, updateSelection).
  • DRY. Lift shared logic into services, directives, pipes, utils, or sub-components.
  • SRP & Rule of One. Split an oversized component into smaller focused ones, each its own file, each ≤ ~400 LoC.
  • Structure. Reorder members to the canonical order; thin the template (logic → computed()/methods, complexity ≤ 10); prefer private, readonly, const.
Show solution – refactor prompt & member order 
promptng-refactor step 6 (Refactor). One concern at a time, show each diff:
1) describe the flow of the big handler, then split it into named functions;
2) extract shared logic into a service/util;
3) if the file is > ~400 LoC, split into focused sub-components;
4) reorder members to the canonical order and thin the template.
Every change must reduce complexity, clarify responsibility, or modernize.

Canonical member order (full details in clean-code-and-structure.md):

decorator (@Component) class (extends & implements) members ├─ injects → input → model → output → queries → signals → other methods └─ static → constructor → other methods
Let AI describe, then cut

The agent is especially strong at function extraction when you ask it to describe the steps first and only then slice them into named functions. Make it narrate before it edits.

Checkpoint: each unit has one responsibility, no file exceeds ~400 LoC, functions are small and named, the template is logic-light, and every diff was gauntlet-green and checkpointed.

7 Review

Goal: confirm the gains, write down the why, and hand the result to a human – you.

Read the finished component top-to-bottom as if you were onboarding to it fresh. The agent's work is treated like a junior's: you review and commit every change.

Your task

Walk the review checklist, summarize what changed and why (the benefit per change), note follow-ups, and only then commit. For a large scope, run a deeper architectural pass.

Show solution – review checklist
  • Reads top-to-bottom; one clear responsibility; canonical member order.
  • No any; small named functions; no file > ~400 LoC; OnPush; no leftover lifecycle hooks.
  • Comments and domain logic intact; tests green and still meaningful (behavior, not internals).
  • Every change delivered a real gain – less complexity, clearer responsibility, or modern.

For a deeper pass on a large scope, use ng-review-architecture. Then commit through your Git GUI, reviewing the diff one last time.

The human commits

The skill never stages or commits for you. Review the diff, then commit it yourself – this is the final guardrail that keeps AI a contributor, not an unchecked author.

Checkpoint: you've reviewed the full diff, the gauntlet is green, the gains are written down, and the change is committed by you.

Definition of Done

Your component is refactored when…

  • ✅ You drove it through all seven steps, each gauntlet-green and checkpointed.
  • ✅ No any, no replaceable lifecycle hooks, no *ngIf/@Input()/ngClass – strict types, signals, inject(), control flow, OnPush instead.
  • ✅ Functions are small and named; no file exceeds ~400 LoC; the template is logic-light.
  • ✅ Comments and domain logic are intact; behavior is unchanged and still covered by tests.
  • ✅ You reviewed every change and committed it yourself, with a note on the gains.

The gauntlet & guardrails

shell# the gauntlet – run after every step
pnpm build && pnpm lint --fix && pnpm format && pnpm test

In a normal project, do not proceed until this is green. In a deliberately dusty fallback, the Analyse step may start red; record that baseline and make every later checkpoint green or no worse than the documented failures.

  • Treat the agent like a junior – review every change before commit.
  • No big-bang rewrite – small, reviewable intermediate steps only.
  • Every change earns its keep – less complexity, clearer responsibility, or modernization.
  • Preserve meaning – never delete existing comments or domain logic; behavior stays the same.
  • Plan first, then implement – ask clarifying questions before editing.
  • Git discipline – clean tree to start; checkpoint per step; you commit, not the agent.
