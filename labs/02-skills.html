Working with Agent Skills

Skills turn an agent from an improviser into a specialist: a folder of instructions it loads on demand and follows the same way every time. In this lab you'll first put a few of this workspace's skills to work on your own project, then author a brand-new skill for something you do over and over.

Angular v22 Focus Agent Skills Est. time 30–45 min Level intermediate Builds on Labs 00–01

About this lab

This is Lab 02. By now you have a project of your own (Lab 00) and the agentic tooling in place (Lab 01). Skills are the next layer: reusable, version-controlled capabilities your agent can invoke. The lab has two parts – use a few skills, then build one of your own.

  • Part 1 – consume. Try two or three of the dozen-plus skills in this repo on your own project.
  • Part 2 – create. Author a custom skill for a repeatable process, using the create-a-skill meta-skill.
Why this matters

A good prompt helps once; a good skill helps every time. Encoding a repeated process as a skill makes the agent's work repeatable, reviewable, and shareable – the same reason you write a script instead of retyping commands. Using a handful of skills teaches you what "good" feels like; writing one teaches you how it works.

Background – what is a skill?

A skill is a folder under .agents/skills/ containing a SKILL.md – the instructions the agent follows – and, optionally, bundled reference files or scripts. The agent keeps each skill's name and description in view and loads the full SKILL.md only when your request matches, so the right process kicks in without you spelling it out. Browse the full catalogue in SKILLS.md: a dozen-plus skills split into custom ones authored here and third-party ones adapted from public sources.

Learning objectives

  • Read a skill's frontmatter and body to judge whether it fits a task.
  • Make a skill available to your agent inside your own project and invoke it.
  • Compare two or three skills and tell when a defined process beats an ad-hoc prompt.
  • Author a new skill – frontmatter, a triggering description, and a tight body.
  • Document and index it so it's discoverable and reusable.

Prerequisites

  • Finished Lab 00 – you have a small project of your own and a written goal.
  • Finished Lab 01 – agentic tooling and an AI coding agent are working.
  • This ng-agentic repo checked out locally, so you can read and copy its skills under .agents/skills/.
How to work through this lab

Same rhythm as the other labs: each step has a Goal, a Your task box, an optional collapsible Solution, and a Checkpoint. Both parts are open-ended – the deliverables are your project moved forward by a few skills, and your new skill. There's no single right answer.

Part 1 – Use a few skills on your project

1 Pick two or three skills that fit your project

Goal: choose two or three skills whose jobs match real tasks in the project you brought to the workshop.

Open SKILLS.md and skim the catalogue. Match skills to the goal you wrote in Lab 00 – not the other way around. Two or three is plenty; you're sampling the toolbox, not exhausting it.

Your task

  • Browse SKILLS.md and open the SKILL.md of a few candidates.
  • Pick two or three that map to concrete tasks on your project.
  • Jot down which task each one will tackle.
Show solution – matching skills to tasks
If a task on your project is… Try
Build a component / feature the idiomatic way angular-developer
Add a form with validation ng-forms
Make it accessible / pass axe ng-accessibility
Modernize old code to signals / standalone ng-migrate
Speed up load or runtime ng-performance
Review the architecture ng-review-architecture
Write a browser e2e test create-e2e-tests
Harden against XSS / CSP issues ng-security

Pick a spread that tells a story – e.g. build something with angular-developer, then ng-accessibility and create-e2e-tests to harden and test it.

Checkpoint: you've named two or three skills and the specific task on your project each one will tackle.

2 Make them available to your agent

Goal: get your chosen skills in front of your agent inside your own project.

A skill is just a portable folder. To use ones from this repo in your project, copy them into the place your agent looks for skills.

Your task

Copy your two or three chosen skill folders into your project's skills directory, then restart your agent so it picks them up.

Show solution – copy the skills in 
shell# from your own project's root – adjust the path to where you cloned ng-agentic
# Claude Code reads skills from .claude/skills/ ; this repo's tool-agnostic home is .agents/skills/
mkdir -p .claude/skills
cp -R ../ng-agentic/.agents/skills/angular-developer  .claude/skills/
cp -R ../ng-agentic/.agents/skills/ng-accessibility   .claude/skills/
cp -R ../ng-agentic/.agents/skills/create-e2e-tests   .claude/skills/

Restart the agent so it re-reads the skill catalogue, then confirm it can see them.

Where skills live depends on your agent

Claude Code discovers skills in .claude/skills/ (project) or ~/.claude/skills/ (personal); other agents differ. This repo keeps a tool-agnostic copy in .agents/skills/. If a skill doesn't trigger, check it's in a directory your agent actually scans, and restart the agent.

Checkpoint: your chosen skills are in your project's skills directory and your agent can see them.

3 Run them as quality gates and compare

Goal: invoke each skill on its task and notice how a defined process changes the result. Skills like accessibility audits and e2e tests act as repeatable quality gates – the same checks run the same way every time.

Trigger each skill by describing the task in its own words – the matching description is what makes it fire. Then look at how it worked, not just what it produced.

Your task

  1. Invoke each of your two or three skills on the task you picked for it.
  2. Let each one drive – answer its questions, but let it follow its own process.
  3. For each, note one thing it did that you'd have forgotten to ask for.
Show solution – triggering a skill

Describe the job the way the skill advertises it, one request at a time:

textBuild a booking-list component following the project conventions.
Now make that component accessible and pass an axe audit.
Write a Playwright e2e test for the booking list.

Each request matches a different skill's description, so the right process kicks in for each.

Tip – read the body alongside the run

Open each skill's SKILL.md while you review the agent's transcript. Seeing the instructions next to the behaviour is the fastest way to learn what makes a skill effective – useful ammunition for Part 2.

Checkpoint: each chosen skill ran on your project, and you can name what its process gave you over a bare prompt.

Part 2 – Build your own skill

4 Find a process worth capturing

Goal: pick one task you do repeatedly, or a custom process with definite steps, to turn into a skill.

The best first skill is small and real: a chore you've explained to an agent more than once, or a team convention you keep restating. If you do it on autopilot, it's a candidate.

Your task

Write down, in plain language:

  • Trigger – when should the agent reach for this? (this becomes the description)
  • Process – the ordered steps you take, including the "always / never" rules.
  • Done – how you know the task is finished.
Show solution – example skill ideas
  • conventional-commit – turn a diff into a Conventional Commits message following the repo's git style guide.
  • new-feature-scaffold – generate a feature folder (component + route + test) the way your team structures them.
  • pr-checklist – review a branch against your own "definition of done" before opening a PR.
  • release-notes – summarize merged PRs since the last tag into changelog entries.

Notice the shape: a clear trigger, a short repeatable process, and an obvious finish line.

Checkpoint: you have a one-paragraph description of a repeatable process, with its trigger and steps.

5 Author the skill

Goal: create a working SKILL.md – frontmatter plus a tight body – using the create-a-skill meta-skill.

Let the repo's own create-a-skill skill guide you: it knows about triggering descriptions, leading words, and keeping the body to the steps that earn their place. Push optional detail into a bundled reference file only if you truly need it.

Your task

  1. Invoke create-a-skill and have it scaffold .agents/skills/<your-skill>/SKILL.md.
  2. Write a description that names the triggers – the words you'd actually say.
  3. Write the body as numbered steps, each ending in a clear "done" condition.
  4. Add the project's frontmatter convention (license + metadata.author).
Show solution – a minimal SKILL.md skeleton 
markdown---
name: conventional-commit
description: Write a Conventional Commits message for the staged changes, following the
  repo's git style guide. Use when the user asks for a commit message, says "commit this",
  or wants to format a commit subject/body.
license: MIT
metadata:
  author: Your Name
  version: '1.0'
---

# Conventional commit

Turn the staged diff into one Conventional Commits message.

## Process

### 1. Read the staged changes
Run `git diff --staged`. If nothing is staged, stop and say so.

### 2. Classify
Pick the type (feat, fix, docs, refactor, test, chore) and an optional scope from the paths touched.

### 3. Write the message
Imperative subject under 72 chars; add a body only if the "why" isn't obvious. Never invent
changes that aren't in the diff.

_Done when_ a valid `type(scope): subject` message is proposed for the user to commit.

That's a complete skill – body only, no bundled files. Add a reference file (e.g. references/examples.md) only when the body would otherwise bloat.

Tip – the description is the most important line

A skill only helps if it triggers. Front-load the description with the words a user would actually use, and say when not to use it. Vague descriptions are why skills sit unused.

Checkpoint: a SKILL.md exists under .agents/skills/<your-skill>/ with valid frontmatter and a numbered body.

6 Make it trigger, then document it

Goal: confirm the skill fires on a natural request, refine it, and add it to the index so it's discoverable.

Your task

  1. Start a fresh agent session and make a request in your own words – does the skill trigger?
  2. If it doesn't, sharpen the description's leading words and try again.
  3. Run update-skills-directory to validate the new SKILL.md and link it from SKILLS.md.
Show solution – verify & index

Test triggering by describing the task, not by naming the skill – that's how it'll be used for real: 

textCommit this for me with a proper message.

Then keep the docs honest:

textUse the update-skills-directory skill to validate and index my new skill.

It checks the SKILL.md and support-file structure, then links your skill into SKILLS.md – the same index you browsed in Step 1.

Iterate – first drafts rarely trigger

If the agent ignores your skill, the description is almost always the culprit. Add the exact phrases you'd say, and a "use when…" clause. Triggering reliability is a wording problem, not a logic problem.

Checkpoint: a natural request triggers your skill, and it has a SKILL.md linked from SKILLS.md.

Definition of Done

You've completed Lab 02 when…

  • ✅ You ran two or three skills from this repo on your own project and saw each follow its process.
  • ✅ You can explain what a skill's process gave you over a plain prompt.
  • ✅ You authored a new skill for a repeatable process, with valid frontmatter and a numbered body.
  • ✅ A natural request triggers your skill (you tuned the description until it did).
  • ✅ Your skill has a SKILL.md and is linked from SKILLS.md.

Next steps

Keep going

Skills compound. Promote the conventions you keep repeating into skills, share them with your team by committing them under .agents/skills/, and revisit SKILLS.md whenever you add one. For the anatomy of a great skill – leading words, information hierarchy, branches – study create-a-skill.
